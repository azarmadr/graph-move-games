## Game State Model (Invariants must hold)

### Board and Moves
- Board is **4x4** (16 cells).
- Cell values are `0` for empty or powers of two (standard 2048 values).
- Moves are single-direction: **Up/Down/Left/Right**.
- A move is **valid** if it results in at least one tile movement or merge.
- Score impact from merges must follow standard 2048 rules.

## Graph Semantics (DAG Global Structure)

### Per-Game Timeline
- Each **game instance** progresses as a **single linear timeline** from its start snapshot to its current snapshot.
- Within a game instance:
  - the active cursor advances forward in time after valid moves
  - nodes are **never repeated** within that game instance (no cycles; no revisiting earlier snapshots)

### Global Graph
- The global graph across all game instances is a **DAG** (directed acyclic graph).
- Nodes **may have multiple incoming edges** (i.e., multiple branches from different games can converge onto the same board-snapshot node).
- The implementation must prevent cycles so the overall structure remains acyclic.

## Nodes
- A node represents a **complete board snapshot** (4x4 array).
- Nodes must have stable identifiers so the UI can reference them.

## Edges
Two conceptual edge types:
- **Move edge:** from the pre-move snapshot node to the post-move snapshot node (merge resolution outcome).
- **Spawn edge:** from a post-move snapshot node to the next snapshot after spawning one configured tile (or more than one spawn step if you model multi-tile spawning as multiple transitions).

Each edge must carry enough metadata for UI labeling/coloring, including at minimum:
- a **game instance identifier**
- for **move edges:** move direction
- for **spawn edges:** spawned tile position and value

## Score Handling (Performance + clarity)
- Do **not** require expensive recomputation by traversing the graph for each UI update.
- Score must be treated as **game-instance state**:
  - the WASM layer maintains cumulative score per game instance
  - the UI displays the score provided by WASM for the active game instance

## Random Tile Generation Rules (Correctness + DAG-friendly branching)
After each **valid** player move:
- Spawn **1 or 2** tiles (configurable; default **1** on 4x4).
- Each spawned tile:
  - uses standard probability: **90% value 2**, **10% value 4**
  - spawns into a **currently empty** cell

Branching guideline:
- If the game reaches a board snapshot node that already exists elsewhere in the global DAG (i.e., that snapshot is reachable via multiple incoming edges), the randomizer should select spawn outcomes that create **distinct outgoing spawn transitions** in the context of the current predecessor/situation, rather than collapsing into an identical already-existing continuation.
- “Distinct outgoing spawn outcome” must be interpreted as at least one of:
  - different spawn position(s)
  - different spawn value(s)
  - (or, for multi-tile spawns) a distinct combined outcome across the full spawn sequence.

## UI Rendering Requirements

### Board Canvas
- Render a 4x4 grid with readable tile values.
- Use recognizable 2048-like colors for tile values (2/4/8/...).
- Display the current score for the active game instance.
- Invalid moves must not change board, score, or graph.

### Graph Canvas (Later Phases)
- Render a global visualization of nodes and edges for all games.
- Color-code edges by game instance.
- Support filtering/highlighting:
  - show only selected game
  - highlight selected game while dimming others
