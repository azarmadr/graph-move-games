Build a 2048 game

- JS + Canvas: render board, handle arrow keys, call WASM

Rules:

Non-negotiable commitments:
- UI payload = full state every move.
- Spawning follows [[model.md]]: one deterministic spawn outcome per valid move (merge → spawn), using the configured `SpawnConfig`. The current implementation already does this; the earlier "enumerate all outcomes / DAG branching" requirement is retired in favor of the model.md contract.
- DAG acyclic by construction: edges added by `extend_path` go from the current board state to the merged board state, then to the spawned board state. Node IDs are content-addressed hashes of the board, not monotonically increasing integers; convergence is still guaranteed by construction.

## UI ↔ WASM contract (minimal)
- JS sends: `{ game_id, direction }`
- WASM returns:
  - `{ active_game_id, game: { game_id, source_node_id, current_node_id, score, is_terminated }, active_board: Board, graph: { nodes, edges } }`
  - No separate “incremental graph” required; full state is returned every time.

UI requirements:
- Board canvas: always render N×M grid + readable tile values + 2048-like colors; invalid moves show no board/score change.

## Current State (implemented)

- **Project setup**: Rust WASM scaffold (`wasm-game`) + Vite React frontend + build pipeline (`pnpm run build-wasm`, `pnpm run dev`).
- **Data model & DAG contract**: `Node`, `Edge`, `Board`, `GameInstance`, `GraphDelta`, `GraphSnapshot` in Rust. u64 IDs are serialized as strings over the JSON bridge.
- **Core move logic**: deterministic merge-only resolution for arbitrary board dimensions, with correct merge scoring and no double-merges.
- **Graph update rules**: `extend_path` creates two edges per valid move (`Move` then `Spawn`), deduplicates nodes via content-addressed hashes, and checks termination after the spawn step.
- **Game instances**: multiple independent `game_id`s, each with `source_node_id`, `current_node_id`, `score`, and `is_terminated`.
- **WASM ↔ JS bridge**: `create_game_with_config`, `make_move`, `get_state`, `export_graph`, `import_graph` all wired and tested.
- **Board rendering**: canvas board with 2048-like colors, keyboard arrows, WASD, on-screen buttons, and touch swipes.
- **Graph rendering**: focused local graph showing the current node, its merge parent, and ancestor nodes.
- **Persistence**: full `ExportData` plus active `game_id` saved to `localStorage`; restored on load via `import_graph` + `get_state`.
- **Export/import UI**: clipboard export/import buttons in the frontend.
